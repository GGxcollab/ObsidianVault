A Organização Internacional de Normalização (ISO) e a Comissão Eletrotécnica Internacional (IEC) criaram a norma ISO/IEC 19249. Nesta tarefa, abordaremos brevemente a ISO/IEC 19249:2017 _Tecnologia da informação - Técnicas de segurança - Catálogo de princípios arquitetônicos e de projeto para produtos, sistemas e aplicações seguros_ . O objetivo é obter uma melhor compreensão dos princípios de segurança recomendados por organizações internacionais.

A norma ISO/IEC 19249 lista cinco princípios _arquitetônicos :_

1. **Separação de Domínios** : Cada conjunto de componentes relacionados é agrupado como uma única entidade; os componentes podem ser aplicativos, dados ou outros recursos. Cada entidade terá seu próprio domínio e receberá um conjunto comum de atributos de segurança. Por exemplo, considere os níveis de privilégio do processador x86: o kernel do sistema operacional pode ser executado no anel 0 (o nível mais privilegiado). Em contraste, os aplicativos em modo de usuário podem ser executados no anel 3 (o nível menos privilegiado). A separação de domínios está incluída no Modelo de Goguen-Meseguer.
2. **Estrutura em camadas** : Quando um sistema é estruturado em vários níveis ou camadas abstratas, torna-se possível impor políticas de segurança em diferentes níveis; além disso, torna-se viável validar a operação. Consideremos o modelo OSI (Interconexão de Sistemas Abertos) com suas sete camadas em redes. Cada camada no modelo OSI fornece serviços específicos para a camada acima dela. Essa estrutura em camadas possibilita a imposição de políticas de segurança e a fácil validação de que o sistema está funcionando conforme o esperado. Outro exemplo do mundo da programação são as operações de disco; um programador geralmente usa as funções de leitura e gravação de disco fornecidas pela linguagem de programação de alto nível escolhida. A linguagem de programação oculta as chamadas de sistema de baixo nível e as apresenta como métodos mais amigáveis ​​ao usuário. A estrutura em camadas está relacionada à Defesa em Profundidade.
3. **Encapsulamento** : Na programação orientada a objetos (POO), ocultamos as implementações de baixo nível e impedimos a manipulação direta dos dados em um objeto, fornecendo métodos específicos para essa finalidade. Por exemplo, se você tiver um objeto relógio, você forneceria um método `increment()`em vez de dar ao usuário acesso direto à `seconds`variável. O objetivo é evitar valores inválidos para suas variáveis. Da mesma forma, em sistemas maiores, você usaria (ou até mesmo projetaria) uma Interface de Programação de Aplicativos ( API ) adequada que seu aplicativo usaria para acessar o banco de dados.
4. **Redundância** : Este princípio garante disponibilidade e integridade. Existem muitos exemplos relacionados à redundância. Considere o caso de um servidor com duas fontes de alimentação integradas: se uma fonte de alimentação falhar, o sistema continua funcionando. Considere uma configuração RAID 5 com três discos: se um disco falhar, os dados permanecem disponíveis usando os dois discos restantes. Além disso, se os dados forem alterados incorretamente em um dos discos, isso será detectado pela paridade, garantindo a integridade dos dados.
5. **Virtualização** : Com o advento dos serviços em nuvem, a virtualização tornou-se mais comum e popular. O conceito de virtualização consiste em compartilhar um único conjunto de hardware entre múltiplos sistemas operacionais. A virtualização oferece recursos de sandbox que aprimoram os limites de segurança, a execução segura e a detecção de programas maliciosos.

A norma ISO/IEC 19249 ensina cinco princípios _de projeto :_

1. **Princípio do Menor Privilégio** : Você também pode expressá-lo informalmente como "princípio da necessidade" ou "princípio da necessidade de saber", ao responder à pergunta "quem pode acessar o quê?". O princípio do menor privilégio ensina que você deve fornecer a quantidade mínima de permissões para que alguém execute sua tarefa, e nada mais. Por exemplo, se um usuário precisa visualizar um documento, você deve conceder a ele permissão de leitura, sem permissão de gravação.
2. **Minimização da Superfície de Ataque** : Todo sistema possui vulnerabilidades que um atacante pode explorar para comprometê-lo. Algumas vulnerabilidades são conhecidas, enquanto outras ainda não foram descobertas. Essas vulnerabilidades representam riscos que devemos buscar minimizar. Por exemplo, uma das etapas para fortalecer um sistema Linux é desabilitar qualquer serviço desnecessário.
3. **Validação Centralizada de Parâmetros** : Muitas ameaças decorrem da entrada de dados recebida pelo sistema, especialmente por parte dos usuários. Entradas inválidas podem ser exploradas para causar vulnerabilidades no sistema, como ataques de negação de serviço e execução remota de código. Portanto, a validação de parâmetros é uma etapa essencial para garantir o estado correto do sistema. Considerando a quantidade de parâmetros que um sistema manipula, a validação desses parâmetros deve ser centralizada em uma única biblioteca ou sistema.
4. **Serviços de segurança gerais centralizados** : Como princípio de segurança, devemos buscar centralizar todos os serviços de segurança. Por exemplo, podemos criar um servidor centralizado para autenticação. Naturalmente, é importante tomar as medidas adequadas para garantir a disponibilidade e evitar a criação de um ponto único de falha.
5. **Preparando-se para o Tratamento de Erros e Exceções** : Sempre que construímos um sistema, devemos levar em consideração que erros e exceções ocorrem e ocorrerão. Por exemplo, em um aplicativo de compras, um cliente pode tentar comprar um item fora de estoque. Um banco de dados pode ficar sobrecarregado e parar de responder a um aplicativo web. Este princípio ensina que os sistemas devem ser projetados para serem à prova de falhas; por exemplo, se um firewall falhar, ele deve bloquear todo o tráfego em vez de permitir todo o tráfego. Além disso, devemos ter cuidado para que as mensagens de erro não vazem informações que consideramos confidenciais, como o despejo de conteúdo da memória que contenha informações relacionadas a outros clientes.
